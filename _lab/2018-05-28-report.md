# Fuzzer

Finishing up algorithms and doing actual implementation of multi-var fuzzer: generator of Julia-types with several type variables. There are two parts that required thinking.

First, having a quasi-type (the type with holes in it), how to poke in type variables in somewhat smart, not insanely redundant way? Consider `Tuple{HOLE, HOLE}`. What are the possible ways to fill in the holes with type variables? Naive strategy suggests: if there are two holes, let's book two fresh variables and try all combinations. In that way we have: `Tuple{T,T}`, `Tuple{T,S}`, `Tuple{S,T}`, `Tuple{S,S}`. But exactly half of these is redundant. One can't distinguish between `Tuple{T,T}` and `Tuple{S,S}` no matter how you do the augmentation step, that is, binding the variables.

So I devised an algorithm to enumerate non-equivalent types. I'm not aware of any standard combinatorial object that corresponds to this.

Second, having a type with free variables (an output from the previous step), how to bind all free variables in various ways? Again, naive strategy would produce too many various placements of where-nodes. The algorithm I came up with takes into account that where-nodes commute, and also can be dragged out of tuples and unions. So I try adding a where-node for a variable only once between a pair of invariant nodes in the type-tree. And the order of where-nodes is the same every time, because `t where T where S` is always equivalent to `t where S where T`.

#Sites
I've been thinking about reasonable way to synchronize PRL-PRG and ERC sites. That required some investigation of Jekyll, the site generator which we use for the sites, which seems to be quite flexible. Also the fact that we use Github Pages for hosting should be taken into account.

Basically, we have two problems. First is sharing the information. It seems that sharing can be done through factoring out relevant data into a git submodule. FAQ suggests that GH Pages understands submodules. But this has to be checked. Second, we need a flexible way of storing publication records that allows for conditional inclusion in the web page. E.g. Konrad has a publication that should probably be on the lab site, but that doesn't mention CTU or ERC Grant, so probably should be excluded from the ERC site. It seems to me that the current way of storing publications in one giant file is not flexible (and also, not extremely convenient) enough to handle this scenario. I think, it is better to use mechanism which Jekyll uses for blog posts: separate .md-files which can be included all on one page via Jekyll-based loop with possible filtering on "tags" that can be added to the "Front Matter", special header in a text file which is used to bring metadata to Jekyll.

Actual implementation if these ideas needs some experimentation.
